**Design Safety: 1/10** — 5 CRITICAL, 2 HIGH, 1 MEDIUM. (Score floors at 1; this program is
trivially drainable as written.)

## What this program does (and where the risk is)

A single-PDA vault that custodies one SPL token: users deposit, and `withdraw` sends tokens back,
signed by the vault PDA. *All* of the risk is concentrated in one place — `withdraw` decides who
gets the vault's money, and right now it asks nobody for permission and trusts every account it's
handed. That's the whole ballgame.

---

## Lesson 1 — Unauthenticated privileged instruction · CRITICAL

**The concept.** On Solana the *caller* chooses which accounts your instruction receives. A pubkey
in an account slot proves nothing — only a **signature** proves the caller controls that key, and
only a binding (`has_one`) proves it's the *right* key for *this* resource.

**In your code.** `programs/vault/src/lib.rs`, `Withdraw`:
```rust
/// CHECK: intentionally unvalidated for the demo
pub authority: UncheckedAccount<'info>,   // not a Signer, not bound to vault.authority
```

**What breaks if unfixed.** I call `withdraw(amount = entire_balance)` and pass *your* vault's
pubkey as `authority`. Nothing requires `authority` to sign, and nothing checks it equals
`vault.authority`. The vault PDA happily signs the token transfer out — to **my** wallet. The
vault is empty before you finish your coffee.

**Real precedent.** Authentication failures are the #1 Solana exploit class. **Wormhole lost
~$326M (Feb 2022)** when its bridge failed to validate the account it used to verify signatures —
a forged account defeated the whole signature check. ([source](https://www.halborn.com/blog/post/explained-the-wormhole-hack-february-2022))

**The fix.**
```rust
#[account(mut, has_one = authority)]
pub vault: Account<'info, Vault>,
pub authority: Signer<'info>,
```
> Principle: every privileged account is a `Signer` **and** bound to its resource.

## Lesson 2 — Unconstrained token accounts · CRITICAL

**The concept.** A `TokenAccount` you don't constrain is just "some token account." You must pin
its **mint** and **authority**, or a caller substitutes one of the wrong mint — or one they own.

**In your code.** `vault_token_account` has no `token::mint` / `token::authority` constraint in
either `Deposit` or `Withdraw`.

**What breaks if unfixed.** I pass a token account of a worthless mint as the "vault," or point
the destination at an account that lets me reroute funds. The program can't tell the difference —
it never said which mint or owner it expected.

**Real precedent.** **Cashio lost ~$52M (Mar 2022)** doing exactly this: it checked collateral
token *types* but never validated the **`mint` field**, so forged collateral minted real
stablecoins. ([source](https://www.halborn.com/blog/post/explained-the-cashio-hack-march-2022))

**The fix.** Move to `InterfaceAccount` (SPL + Token-2022) and constrain:
```rust
#[account(mut, token::mint = mint, token::authority = vault)]
pub vault_token_account: InterfaceAccount<'info, TokenAccount>,
```
> Principle: every token account is pinned to a known mint and authority.

## Lesson 3 — Integer underflow on balances · CRITICAL

**The concept.** Rust **wraps** on overflow in release builds unless `overflow-checks = true`.
Subtracting past zero on a `u64` doesn't error — it returns ~18 quintillion.

**In your code.** `Cargo.toml` ships `overflow-checks = false`, and:
```rust
vault.total_deposited = vault.total_deposited - amount;   // withdraw
```

**What breaks if unfixed.** I withdraw 1 more than the vault holds. `total_deposited` underflows to
a gigantic number, so the accounting now thinks the vault is effectively infinite — every
subsequent withdraw "checks out." Combined with Lesson 1, it's a clean, repeatable drain.

**Real precedent.** *Taught from first principles* — there's no single canonical Solana
post-mortem for a bare-arithmetic underflow (the library has no clean exemplar, so I won't
mis-attribute one). It is, however, the most common finding in Solana audits.

**The fix.**
```rust
// Cargo.toml
[profile.release]
overflow-checks = true
// handler
require!(amount <= vault.total_deposited, VaultError::InsufficientFunds);
vault.total_deposited = vault.total_deposited.checked_sub(amount).unwrap();
```
> Principle: `overflow-checks = true` **and** `checked_*` on every value computation.

## Lesson 4 — Live upgrade authority on a custodial program · HIGH

**The concept.** The upgrade authority can replace your entire program with new code. On a vault
that holds funds, that key *is* the security model — a single hot key is one phishing email from a
total loss.

**In your code.** Not in `lib.rs` — it's a deployment property. The program is upgradeable with no
documented authority custody.

**What breaks if unfixed.** A leaked/socially-engineered upgrade (or admin) key lets an attacker
ship a malicious version or flip privileged config — no code bug required.

**Real precedent.** **Drift lost ~$285M (Apr 2026)** when a weak 2-of-5, zero-timelock Security
Council was social-engineered into pre-signing admin transactions — the contracts were fine, the
*key surface* wasn't. ([source](https://www.chainalysis.com/blog/lessons-from-the-drift-hack/))
(See also Raydium's ~$4.4M admin-key compromise, Dec 2022.)

**The fix.** Hold the upgrade authority in a **Squads Protocol v4** multisig behind a timelock;
publish the policy; plan a path to immutability.
> Principle: privileged keys are multisig + timelock, never a single hot wallet on mainnet.

## The 3 habits that would have prevented most of this

1. **Every privileged account is a `Signer` bound by `has_one`.** (Kills Lesson 1.)
2. **Every account you read is typed and constrained** — owner, mint, authority. (Kills Lesson 2 and most of branch 1.)
3. **Every value computation is `checked_*` with the invariant `require!`d**, and `overflow-checks = true`. (Kills Lesson 3.)

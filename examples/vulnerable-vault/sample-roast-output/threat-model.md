## Detailed Findings

### F-01 ‚Äî withdraw is unauthenticated ¬∑ CRITICAL
- **Where:** `programs/vault/src/lib.rs`, `Withdraw` accounts struct.
- **What:** `authority: UncheckedAccount` gates nothing ‚Äî it never signs and is never compared to
  `vault.authority`. Any caller can invoke `withdraw`.
- **Why it matters:** Direct, trivial drain of the entire vault. The single worst bug here.
- **Fix:** `authority: Signer<'info>` + `#[account(mut, has_one = authority)]` on `vault`.
- **Decision:** accepted.

### F-02 ‚Äî arbitrary CPI via unverified token program ¬∑ CRITICAL
- **Where:** `Withdraw.token_program: UncheckedAccount`, used in `CpiContext::new_with_signer`.
- **What:** The vault PDA signs a CPI into a caller-supplied program.
- **Why it matters:** Attacker substitutes their own program and receives a PDA-signed call.
- **Fix:** `Interface<'info, TokenInterface>` (supports SPL + Token-2022) or `Program<Token>`.
- **Decision:** accepted (Token-2022 support requested ‚Üí Interface).

### F-03 ‚Äî unconstrained token accounts ¬∑ CRITICAL
- **Where:** `deposit.vault_token_account`, `withdraw.vault_token_account`, `withdraw.destination`.
- **What:** No `token::mint` / `token::authority` constraints.
- **Why it matters:** Wrong-mint or attacker-owned token accounts pass validation ‚Üí mis-accounting / theft.
- **Fix:** constrain to the expected mint and the vault PDA authority.
- **Decision:** accepted.

### F-04 / F-07 ‚Äî unchecked math + overflow-checks off ¬∑ CRITICAL
- **Where:** `deposit` (`+`), `withdraw` (`-`); `Cargo.toml [profile.release] overflow-checks = false`.
- **What:** `withdraw` underflows `total_deposited` to a huge u64; release builds wrap.
- **Why it matters:** Accounting can be driven to nonsense; combined with F-01/F-03 it's a clean drain.
- **Fix:** `overflow-checks = true`, `checked_add`/`checked_sub`, and `require!(amount <= total_deposited)`.
- **Decision:** accepted.

### F-05 ‚Äî user-supplied PDA bump ¬∑ HIGH
- **Where:** `initialize(vault_bump: u8)` ‚Üí `vault.bump = vault_bump`.
- **What:** Caller controls the stored bump; a non-canonical valid bump can be supplied.
- **Why it matters:** Enables alternate valid addresses / weakens PDA guarantees.
- **Fix:** use Anchor's bare `bump` at init and `bump = vault.bump` after; drop the argument.
- **Decision:** accepted.

### F-06 ‚Äî upgrade authority undocumented ¬∑ HIGH
- **Where:** program config / deploy (not code).
- **What:** Upgradeable with no stated authority custody plan.
- **Why it matters:** A single hot upgrade key can replace the whole program ‚Äî dominant rug vector.
- **Fix:** Squads Protocol v4 multisig + timelock; publish policy; path to immutability.
- **Decision:** accepted.

### F-08 ‚Äî Token-2022 fee-on-transfer over-credit ¬∑ MEDIUM
- **Where:** `deposit` credits `amount` rather than the received delta.
- **What:** Transfer-fee mints deliver less than `amount`; crediting `amount` over-credits.
- **Why it matters:** Drainable when fee-bearing mints are allowed.
- **Fix:** credit `balance_after - balance_before`, or allow-list non-fee mints.
- **Decision:** accepted.

## Trust Assumptions (residual risk after fixes)

- The upgrade authority is not compromised before immutability.
- Any allowed Token-2022 mint's extensions (hooks/fees) are within the documented policy.
